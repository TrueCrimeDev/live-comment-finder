# Live Comment Finder

A Chrome (Manifest V3) extension that **captures live comments** from the current
page and lets you **search them in real time** from a side panel. Comments stay
searchable even after the site removes them from the DOM. **Everything stays on
your device — nothing is ever transmitted.**

## Features

- Captures comments already visible when monitoring starts, plus new ones as they arrive.
- Retains captured comments after the website virtualizes/removes their DOM nodes.
- Search by **comment text**, **author**, or both — updates as you type (no Enter needed).
- Search modes: case-insensitive substring (default), case-sensitive, whole-word, exact-phrase, and regex (with inline validation for bad patterns).
- Newest-first / oldest-first sorting; result and captured counts.
- Click a result to scroll to and temporarily highlight the original message; a clear fallback message when it is no longer in the page.
- Pause / resume capture, clear history, and a configurable retention cap (1,000–50,000).
- Export results to **JSON** or **CSV** (CSV is injection-safe).
- Keyboard accessible; light/dark themes; reduced-motion aware.

## Supported sites

- **YouTube Live Chat** (including the chat iframe) — built-in adapter.
- **Any other site** — a generic adapter: click **Select Comment Feed**, then click the
  list/container that holds the comments. The choice is remembered per hostname.

## Privacy

- All capture and search happens locally. No analytics, telemetry, tracking, or remote logging.
- Captured comments are **never** sent over the network and are **not** persisted across browser restarts (only your settings persist).
- No remote code, no `eval`, no `innerHTML` for page content — captured text is always rendered as text.

## Requirements

- **Node.js ≥ 20** to build (the build toolchain depends on the `File` global added in Node 20). An `.nvmrc` pins Node 22.
- Google Chrome 116+ (for the Side Panel API).

> **WSL note:** if you develop under WSL on a `/mnt/c` path, install dependencies on the
> Linux filesystem (npm corrupts large native binaries written to the Windows mount). This
> repo expects `node_modules` to resolve normally; if `esbuild` reports `Exec format error`,
> reinstall with the project on the Linux filesystem (or symlink `node_modules` there).

## Install (load the unpacked development build)

1. `npm install && npm run build`
2. Open `chrome://extensions`.
3. Enable **Developer mode** (top right).
4. Click **Load unpacked** and select the generated **`dist/`** folder.
5. Pin the extension and click its toolbar icon to open the side panel.

## Using search

- Type in the **search** box to filter by comment text; type in **Author** to filter by author. Both combine.
- Pick a **mode**: Substring, Case (case-sensitive), Word (whole-word), Phrase (exact), or Regex.
  - Invalid regex shows an inline message instead of failing.
- **Sort** newest/oldest. **Max** sets the retention cap. **Clear filters** resets the query/author/mode.
- **Export JSON / CSV** exports the current result set.

### Keyboard

- `/` or `Ctrl/Cmd + F` — focus the search box.
- `↑` / `↓` — move between results. `Enter` — locate the selected result.
- `Esc` — clear the active search.

## Selecting a generic comment feed

1. On a non-YouTube page, click **Select Comment Feed** in the side panel.
2. Chrome asks for permission to access that site — this is requested **at runtime**, only for that host.
3. Move the pointer to highlight elements; **click** the container that holds the comments. Press **Esc** to cancel.
4. The extension infers the repeated comment-row pattern and starts capturing. Your selection is saved per hostname.

## Permissions explained

| Permission | Why |
| --- | --- |
| `sidePanel` | Show the search UI in Chrome's side panel. |
| `storage` | Save settings only (retention cap, default mode, per-host generic selectors). Never comment content. |
| `activeTab` | Talk to the page you're actively using when you open the panel. |
| `scripting` | Inject the capture script into a page **after you grant a host permission** for the generic adapter. |
| host `*://*.youtube.com/*` | Run the YouTube Live Chat adapter automatically on YouTube. |
| runtime host grants | The generic adapter requests the **current site's** host permission only when you click *Select Comment Feed*. There is **no** `<all_urls>` permission. |

## Known limitations

- Captured comments are not kept across browser restarts (by design; no opt-in persistence yet).
- Third-party sites change their DOM; the YouTube adapter's selectors may need updating if YouTube changes its markup (see the adapter guide).
- A comment removed by the site's virtualization stays **searchable** but can no longer be scrolled-to/highlighted (you'll see the fallback message).
- The generic adapter infers a single repeated row pattern; deeply irregular feeds may need a more specific container.

## Troubleshooting

- **"Feed not found" on YouTube** — the live chat may still be loading or popped out; reload the watch page with chat docked.
- **Generic page shows "Unsupported"** — that's expected until you use *Select Comment Feed*.
- **Nothing captured after selecting a feed** — pick the element that directly contains the repeated comment rows, not a far-outer wrapper.
- **Build fails with `File is not defined` or `Exec format error`** — use Node ≥ 20 and install on the Linux filesystem (see WSL note).

## Architecture

Four cooperating contexts; the **content script** is the source of truth and survives
service-worker suspension via snapshot-on-connect.

```
content script  → capture (one MutationObserver) → dedup → per-tab ring buffer
   │  NEW_COMMENT_BATCH / SNAPSHOT (runtime messaging)
service worker  → opens panel, tracks active tab, runtime-permission generic injection
   │
side panel      → mirror + search/filter/sort + windowed render + export
```

Shared, fully unit-tested pure modules live in `src/shared/` (model, messages+validators,
normalize, dedup, ring buffer, search, export). See
`docs/superpowers/specs/2026-06-20-live-comment-finder-design.md` for the full design.

## Adapter development guide

Adapters are isolated from search and UI. To add one (e.g., Twitch):

1. **Detection** — implement `canHandle(ctx)` (match `ctx.location.hostname`).
2. **Locate feed** — implement `locateFeed(ctx)` returning the smallest container element that holds the comment rows.
3. **Parse messages** — implement `parseComment(node, ctx): ParsedComment[]`. Pull author + text; for mixed content, include emoji `alt` text and link text. Return `[]` for system/UI nodes.
4. **Stable locators** — set `locator` on each `ParsedComment` (prefer a platform message id; it drives dedup via `commentId`). Implement `locateComment(locator, ctx)` so results can be re-found and highlighted.
5. **Fixtures** — add a static HTML snapshot under `test/fixtures/` representing the expected structure (never scrape live DOM in tests).
6. **Tests** — add `test/<adapter>.test.ts` (`// @vitest-environment jsdom`) asserting parsing of normal, mixed-content, and system nodes, plus `locateFeed`.
7. **Register** — add the adapter to the `adapters` array in `src/content/bootstrap.ts`.

Keep all site-specific selectors inside the adapter and document them inline so repairs stay local. Third-party DOM changes may require updating an adapter.
